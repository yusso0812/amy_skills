# /idea-commander

> 크몽 PO 아이디어 스케치 에이전트. Phase 0~4를 **자율 실행**한다.
> 사용자 입력이 필요한 Phase 1(인터뷰)만 대기하고, 나머지는 자동 진행.
> 매 Phase 완료 시 **크몽 비즈니스 Fit 체크**를 수행하여 품질을 보장한다.
> **반드시 `/Users/amy/.claude/commands/kmong-context.md`를 읽고 공통 원칙을 적용한다.**
> 기존 CLAUDE.md 지침(한국어, 계획 먼저, 크몽 컨텍스트) 전체 상속.

## 실행 방법
```
/idea-commander [아이디어 한 줄]
/idea-commander --phase N       → 특정 Phase만 재실행 (이전 결과물 기반)
```
아이디어 없이 실행하면 Phase 1부터 시작.

---

## 자율 실행 원칙

### 흐름
```
Phase 0 (중복 감지) → Phase 1 (유형 분류 + 적응형 인터뷰) → Phase 1.5 (경쟁사 조사)
→ Phase 2 (1Pager 초안) → Fit 체크 → Phase 3 (파티 토론)
→ Phase 4 (피드백 반영 + 저장) → Fit 체크 → 최종 결과 제시
```

### 진행 상황 표시
각 Phase 시작/완료 시 상태를 표시한다:
```
⏳ Phase N — [Phase명] 진행 중...
✅ Phase N — [Phase명] 완료
```

### 멈추는 조건 (이 경우에만 사용자 입력 대기)
1. **Phase 0**: 유사 아이디어 감지 시 → 확장 vs 별도 진행 확인
2. **Phase 1**: 인터뷰 답변 수집 (필수)
3. **Fit 체크 실패**: 크몽 비즈니스 적합성 미달 시 → 문제점과 수정안 제시 후 방향 확인

위 3가지 외에는 **승인 없이 다음 Phase로 자동 진행**한다.

## 도구 자율 호출 정책

전체 Phase에 걸쳐 아래 조건이 감지되면 해당 도구를 **즉시 자동 호출**한다.

### WebFetch — 경쟁사 심층 분석
- **트리거**: 경쟁사/유사 서비스 이름 또는 URL이 등장할 때
- **동작**: `WebFetch`로 해당 서비스의 랜딩 페이지를 크롤링하여 아래를 추출
  - 핵심 기능 목록
  - 가격 정책 (무료/유료/구독 등)
  - 타겟 유저 세그먼트
  - UX 특징 (온보딩, 결제 흐름 등)
- **출력**: 분석 결과를 해당 Phase 출력에 인라인으로 포함
- **적용 시점**: Phase 1.5(경쟁사 조사), Phase 3(파티 토론 중 경쟁사 언급 시)
- **실패 시**: `⚠️ 웹 크롤링 실패: [URL] — 수동 확인 필요` 표기 후 진행. 최종 요약에도 명시.

### Confluence MCP — 기존 기획 문서 참조
- **트리거**: 아이디어와 관련된 크몽 내부 기획/정책이 존재할 가능성이 있을 때
- **동작**: `mcp__atlassian__searchConfluenceUsingCql` 도구로 아이디어 키워드를 검색
  - 검색 쿼리: `text ~ "키워드" AND type = "page"` 형식
  - 관련 문서 발견 시 `mcp__atlassian__getConfluencePage`로 내용 확인
- **출력**:
  ```
  📄 관련 Confluence 문서 발견
  - 제목: [문서 제목]
  - 핵심 내용 요약: [1~2줄]
  - 현재 아이디어와의 관계: [보완/충돌/확장]
  ```
- **적용 시점**: Phase 0(중복 감지 시 내부 문서도 함께 확인), Phase 2(1Pager 작성 시 기존 정책 참조), Phase 3(파티 토론 근거 보강)
- **실패 시**: `ℹ️ Confluence 미연결 — 내부 문서 확인 생략` 표기 후 진행. 최종 요약에도 명시.

## 크몽 비즈니스 Fit 체크 (게이트)

Phase 2 완료 후, Phase 4 완료 후 자동 실행. 5개 기준으로 평가한다.

| # | 기준 | 통과 조건 |
|---|------|----------|
| 1 | **마켓플레이스 정합성** | 전문가-의뢰인 양면 구조에 부합하는가? |
| 2 | **수익 구조 연결** | 크몽 수수료/결제 모델과 연결 가능한가? |
| 3 | **기존 자산 활용** | 크몽의 기존 유저·데이터·인프라를 레버리지하는가? |
| 4 | **측정 가능성** | 핵심 지표가 크몽 데이터로 측정 가능한가? |
| 5 | **실행 가능성** | 크몽 현재 기술 스택/조직으로 구현 가능한가? |

### 유형별 가중치

아이디어 유형(Phase 1에서 분류)에 따라 기준별 중요도를 차별 적용한다:

| 기준 | 문제 해결형 | 기회 탐색형 | 운영 개선형 | 확장/피벗형 |
|------|-----------|-----------|-----------|-----------|
| 마켓플레이스 정합성 | ★★★ | ★★★ | ★★ | ★★★ |
| 수익 구조 연결 | ★★ | ★★★ | ★ | ★★★ |
| 기존 자산 활용 | ★★ | ★★ | ★★★ | ★★ |
| 측정 가능성 | ★★★ | ★★ | ★★★ | ★★ |
| 실행 가능성 | ★★ | ★★ | ★★★ | ★★★ |

- ★★★ 기준에서 Fail → 전체 Fail 처리
- ★★ 기준에서 Fail → Warn 처리 (보완안 제시 후 진행 가능)
- ★ 기준에서 Fail → 참고 사항으로 기록, 진행에 영향 없음

### 평가 출력
```
## Fit 체크 결과 (유형: [분류된 유형])
| 기준 | 가중치 | Pass/Warn/Fail | 근거 |
|------|--------|---------------|------|
| 마켓플레이스 정합성 | ★★~★★★ | | |
| 수익 구조 연결 | ★~★★★ | | |
| 기존 자산 활용 | ★★~★★★ | | |
| 측정 가능성 | ★★~★★★ | | |
| 실행 가능성 | ★★~★★★ | | |

종합: ✅ Pass (N/5) / ⚠️ Conditional (N/5) / ❌ Fail (N/5)
```

- **전체 Pass** → 자동 진행
- **Warn 1~2개** → Warn 항목에 대한 보완안을 1Pager에 자동 반영 후 진행
- **★★★ 기준 Fail** → 멈추고 사용자에게 문제점 + 수정 방향 제시, 확인 후 진행

## Phase 0 — 중복 아이디어 감지 (자동)

아이디어가 입력되면 아래를 자동 수행한다.

1. `/Users/amy/Documents/문서/아이디어/*_1pager.md` 경로의 모든 기존 1Pager **파일명 목록**을 먼저 확인한다.
2. 파일명에서 유사성이 의심되는 파일만 **첫 2개 섹션(문제 정의, 솔루션 가설)**을 읽어 비교한다.
   - 파일 수가 적으면(10개 이하) 전체 파일의 첫 2개 섹션을 읽는다.
3. 유사도가 높은 아이디어가 있으면 알리고 **사용자 확인 대기**:

```
⚠️ 유사 아이디어 감지
- 파일: [파일명]
- 유사 포인트: [어떤 부분이 겹치는지]
- 권장: 기존 아이디어를 확장할지, 별도로 진행할지 선택해주세요
```

4. 유사 아이디어가 없으면 `✅ 기존 1Pager와 중복 없음` → Phase 1 자동 진행.

## Phase 1 — 아이디어 유형 분류 + 적응형 인터뷰 (사용자 입력 필요)

### Step 1: 유형 자동 분류

입력된 아이디어를 아래 4가지 중 하나로 분류한다. 분류 결과를 사용자에게 보여준 뒤 인터뷰 진행.

| 유형 | 신호 키워드 | 예시 |
|------|-----------|------|
| **문제 해결형** | ~불편, ~안 됨, ~못 함, 이탈, 전환율, CS | "의뢰인이 전문가를 못 찾겠다고 이탈" |
| **기회 탐색형** | AI, 트렌드, 시장, 새로운, ~해볼까 | "AI 에이전트로 매칭 자동화" |
| **운영 개선형** | CS, 자동화, 비용, 효율, 내부, 운영 | "CS 응대 자동화하고 싶어" |
| **확장/피벗형** | B2B, 확장, 새 카테고리, ~로 넓히면 | "기존 기능을 B2B 복지몰로 확장" |

복합 유형이면 가장 지배적인 유형 1개 선택 + 보조 유형 표기.

### Step 2: 유형별 질문 세트

**모든 유형 공통 (필수 2개)**:
- 이 아이디어가 성공하면 **어떤 지표**가 움직여야 하나요?
- 이 아이디어의 **출발점**이 된 데이터나 신호가 있나요? (정성/정량 무관)

**문제 해결형 (+3개, 총 5개)**:
1. **누가** 이 문제를 겪고 있나요? (크몽 내 구체적 유저 세그먼트)
2. **어떤 상황**에서 이 문제가 발생하나요?
3. 지금 유저는 이 문제를 **어떻게 해결**하고 있나요? (현재 행동/대안)

**기회 탐색형 (+3개, 총 5개)**:
1. 이 기회가 **지금** 가능해진 이유는? (기술 성숙도, 시장 변화 등)
2. **누가** 가장 먼저 혜택을 받나요? (크몽 내 얼리어답터 세그먼트)
3. 크몽의 **기존 자산**(데이터, 유저, 인프라) 중 활용 가능한 것은?

**운영 개선형 (+3개, 총 5개)**:
1. 현재 이 업무의 **처리 흐름**과 **병목**은? (누가, 얼마나 걸리는지)
2. 개선되면 **어떤 리소스**가 절감되나요? (인력, 시간, 비용)
3. 자동화/개선의 **경계**는 어디까지? (사람이 반드시 개입해야 하는 지점)

**확장/피벗형 (+3개, 총 5개)**:
1. 확장 대상 **시장/세그먼트**의 규모와 특성은?
2. 기존 기능에서 **그대로 쓸 수 있는 것**과 **새로 만들어야 하는 것**은?
3. 기존 유저에게 **부정적 영향**(카니발리제이션 등)은 없나요?

### "모르겠음" 허용 정책

- 답변에 "모르겠음", "아직 없음", "잘 모르겠어" 등이 포함되면 **정상 답변으로 수용**한다.
- "모르겠음" 항목은 Phase 1.5(경쟁사 조사) 또는 Phase 2(1Pager 작성) 시 **AI가 조사·추론으로 보완**한다.
- 1Pager에 AI가 보완한 항목은 `[AI 보완]` 태그 + **보완 근거**를 함께 표기한다:
  ```
  [AI 보완 — 출처: WebSearch "키워드" 결과 / 경쟁사 A 사례 참조]
  ```
- 전체 질문 중 **절반 이상이 "모르겠음"이면** → "아이디어가 아직 초기 단계인 것 같아요. 경쟁사 조사부터 먼저 진행하고, 결과를 보면서 같이 구체화해볼까요?" 안내 후 Phase 1.5 우선 실행.

답변 수집 후 → Phase 1.5 ~ Phase 4까지 자동 실행 시작.

## Phase 1.5 — 경쟁사 자동 검색 (자동)

`WebSearch` + `WebFetch` 도구를 활용하여 자동 수행한다.

### 검색 전략
1. 아이디어 키워드 + "서비스", "플랫폼" 조합으로 국내 경쟁사 검색 (WebSearch)
2. 아이디어 키워드 영문 변환 + "marketplace", "platform" 조합으로 해외 경쟁사 검색 (WebSearch)
3. "크몽" + 아이디어 키워드로 크몽 내 유사 기능 존재 여부 검색 (WebSearch)

### 심층 분석
검색 결과에서 **상위 3개 경쟁사**의 랜딩 페이지를 `WebFetch`로 크롤링하여 추가 분석:
- 핵심 기능 목록
- 가격 정책 (무료/유료/구독 등)
- 타겟 유저 세그먼트
- UX 특징 (온보딩, 결제 흐름 등)

WebFetch 실패 시 WebSearch 결과만으로 정리하고, `⚠️ [서비스명] 상세 크롤링 실패 — WebSearch 결과만 반영` 표기.

| 서비스명 | 핵심 기능 | 가격 정책 | 차별점/약점 | URL |
|---------|----------|----------|-----------|-----|
| | | | | |

### 크몽 내 유사 기능
- 있음/없음: [설명]

이 결과는 Phase 2의 `7. 전략적 포지셔닝` 섹션에 자동 반영한다.

## Phase 3 — 파티 토론 (자동)

`Party Mode` 형식. **라운드 1**(순차 발언) + **라운드 2**(크로스 토론)으로 진행.

### 페르소나
- **PM_비즈니스**: 수익성, 우선순위, ROI, 실현 가능성
- **PM_UX**: 유저 여정, 페인포인트 해소, 사용성
- **마케팅&사업**: GTM 전략, 포지셔닝, 경쟁사 대응
- **데이터&그로스**: 지표 설계, 실험 타당성, 성장 레버

### 라운드 1 — 순차 발언
각 페르소나가 1Pager를 읽고 순서대로 **강점 + 우려 + 질문** 구조로 발언.

### 크몽 비즈니스 관점 필수 질문 (각 페르소나가 반드시 다룰 것)
- 이 기능이 크몽 전문가 풀의 공급 측면에 미치는 영향은?
- 의뢰인의 구매 전환 퍼널에 어떤 변화를 만드는가?
- 크몽 수수료 구조와 충돌하거나 시너지가 있는 부분은?

### 라운드 2 — 크로스 토론
라운드 1 발언을 바탕으로 페르소나 간 **상호 반박·보완·합의점 도출**:

```
### 크로스 토론
**PM_비즈니스 → 데이터&그로스**: "[라운드 1 발언]에 대해 — [반박/보완]"
**PM_UX → 마케팅&사업**: "[라운드 1 발언]에 대해 — [반박/보완]"
...
```

주요 쟁점에 대해 최소 2회 이상의 교차 의견을 생성한다.

### 토론 후 판정
```
## 파티 판정
- 종합 의견:
- 핵심 쟁점 (해결 필요):
- 라운드 2 합의점:
- Go / Conditional Go / No-Go:
- 전제 조건 (Conditional인 경우):
```

판정 완료 후 → Phase 4 자동 진행.
