# 페르소나 토론

기획 내용을 다각도로 검증하기 위해, AI 페르소나 5명이 구조화된 토론을 진행하고 실행 가능한 보고서를 산출한다.

## 핵심 원칙

- **"GPT한테 피드백 달라고 한 것"이 아니라, "5명의 실제 이해관계자가 회의실에서 부딪힌 것"에 가까운 경험**을 제공한다
- 페르소나가 진짜 사람처럼 느껴져야 하고, 토론에서 실제 충돌이 발생해야 한다
- 합의가 아니라 **충돌을 통한 발견**이 목적이다

---

## 실행 파이프라인

11개 STEP을 순서대로 진행한다. 각 STEP에서 사용자 입력이 필요한 경우 반드시 기다린다.

```
STEP 1 맥락 질문 (사용자 입력)
STEP 2 기획 내용 (사용자 입력)
STEP 3 검증 목적 (사용자 입력)
STEP 4 걱정 포인트 (사용자 입력)
STEP 5 기획서 구조화 + 보완 (시스템 + 사용자)
STEP 6 시장 규모 분석 (시스템 자동)
STEP 7 페르소나 확인 (사용자 확인)
STEP 8 토론 진행 (시스템)
STEP 9 1:1 심층 인터뷰 (사용자 확인 후)
STEP 10 보고서 확인 (사용자 확인)
STEP 11 보고서 생성 (시스템)
```

---

## STEP 1 — 맥락 질문

사용자에게 두 가지를 물어본다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 1/11 — 기획 맥락
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 Q1. 기획 단계가 어디인가요?

   (a) 아이디어 단계 — 방향 자체를 탐색 중
   (b) 구체화 중 — 큰 그림은 있고 디테일 잡는 중
   (c) 거의 확정 — 빠진 거 없는지 최종 점검
   (d) 개발 진행 중 — 피봇/수정 여부 판단

 Q2. 기획 유형은 무엇인가요?

   (a) B2C 서비스/앱
   (b) B2B SaaS/솔루션
   (c) 내부 도구/시스템
   (d) 콘텐츠/미디어
   (e) 하드웨어/IoT
   (f) 기타: ___
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

기획 단계에 따라 이후 토론 모드를 자동 선택한다:

| 기획 단계 | 기본 토론 모드 | 보고서 초점 |
|----------|---------------|------------|
| 아이디어 | `standard` | "이 방향이 맞는가?" |
| 구체화 중 | `standard` | "빠진 게 뭔가?" |
| 거의 확정 | `red-blue` | "뚫리는 데가 없는가?" |
| 개발 진행 중 | `premortem` | "계속 갈 것인가, 바꿀 것인가?" |

기획 유형에 따라 페르소나 프리셋을 자동 선택한다:

| 기획 유형 | 기본 프리셋 |
|----------|-----------|
| B2C | `b2c` |
| B2B | `b2b` |
| 내부 도구 | `internal` |
| 기타 | `auto` (기획 내용 분석 후 자동 구성) |

 STEP 2/11 — 기획 내용
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 기획 내용을 입력해주세요.

 입력 방법:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

한 줄 아이디어부터 상세 기획서까지 모두 수용한다. 부족한 부분은 STEP 5에서 보완한다.

## STEP 3 — 검증 목적

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 3/11 — 검증 목적
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 이 토론에서 가장 알고 싶은 건 무엇인가요? (복수 선택 가능)

   (a) 만들 가치가 있는가 (시장성/필요성)
   (b) 사용자가 실제로 쓸까 (사용성/UX)
   (c) 돈이 되는 구조인가 (수익성/비즈니스 모델)
   (d) 기술적으로 가능한가 (기술 타당성)
   (e) 뭘 먼저 만들어야 하나 (우선순위)
   (f) 경쟁에서 이길 수 있나 (차별화/포지셔닝)
   (g) 직접 입력: ___
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

선택된 목적에 따라 토론의 초점과 페르소나 발언 비중이 달라진다:

| 검증 목적 | 강조되는 관점 | 발언 가중치 |
|----------|-------------|-----------|
| 시장성 | "이 문제가 진짜 존재하는가?" | S5 회의론자 강화 |
| 사용성 | "실제 쓰는 장면이 그려지는가?" | S1, S2 사용자 강화 |
| 수익성 | "돈을 낼 이유가 있는가?" | S4 전문가 + S5 회의론자 강화 |
| 기술 타당성 | "현실적 제약 안에서 가능한가?" | S4 전문가 강화 |
| 우선순위 | "MVP에 뭘 넣고 뭘 뺄까?" | 전원 균등 |
| 차별화 | "왜 기존 것 대신 이걸 쓰나?" | S5 회의론자 + S3 인접사용자 강화 |

## STEP 4 — 걱정 포인트

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 4/11 — 걱정 포인트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 이미 걱정되는 부분이 있나요? (없으면 건너뛰기)

 예시:
   "수익 모델이 아직 약한 것 같다"
   "타겟이 너무 좁지 않을까"
   "비슷한 서비스가 이미 있는데 차별점이 약하다"

 자유롭게 입력해주세요 (여러 개 가능):
 ___

 (건너뛰려면 Enter)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

걱정 포인트 처리 원칙:
- 토론 쟁점으로 반드시 1회 이상 다룬다
- "네, 약합니다"로 끝내지 않는다 — **"어떻게 할 수 있는가"** 또는 **"생각보다 심각하다/괜찮다"**까지 도달한다
- 사용자가 건너뛴 경우 → 기획서 분석 기반으로 잠재 리스크를 자체 식별한다

## STEP 5 — 기획서 구조화 + 보완 질문

STEP 2의 기획 내용을 다음 구조로 정규화한다:

```yaml
planning_structure:
  problem:           # 어떤 문제를 해결하는가
    target_pain:     # 타겟의 핵심 페인포인트
    current_alt:     # 현재 대안은 무엇인가
  solution:          # 어떻게 해결하는가
    core_feature:    # 핵심 기능 (최대 3개)
    differentiator:  # 차별점
  target:            # 누구를 위한 것인가
    primary_user:    # 핵심 타겟
    context:         # 사용 맥락
  business:          # 어떻게 돈을 버는가
    model:           # 수익 모델
    scale:           # 규모 가정
  constraints:       # 제약 조건
    tech / resource / regulation
```

정보가 충분하면 → "이렇게 이해했습니다. 맞나요?" 확인 후 STEP 6으로 진행한다.

부족하면 → **최대 5개** 보완 질문을 한다:
- STEP 1~4에서 이미 답변된 내용은 중복 질문하지 않는다
- 없어도 토론 가능한 항목은 빈칸으로 두고 진행한다
- "모르겠다"고 답하면 → 그 자체를 토론 쟁점으로 등록한다

## STEP 6 — 시장 규모 분석

시스템이 자동으로 수행한다. 사용자에게 추가 입력을 요구하지 않는다.

STEP 5의 구조화된 기획을 기반으로 분석하고 표 형태로 간결하게 출력한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 6/11 — 시장 규모 분석
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 | 구분 | 규모 | 산출 근거 |
 |------|------|----------|
 | TAM | ~5조원 | 국내 외식 O2O 시장 |
 | SAM | ~8,000억원 | 직장인 점심 추천 세그먼트 |
 | SOM | ~200억원 | 수도권 오피스 밀집지 초기 타겟 |

 경쟁: 배민(사용자 기반↑, 배달 중심↓) · 네이버지도(리뷰DB↑, 개인화↓)
 트렌드: 점심 선택 피로 증가, 개인화 추천 수요 성장
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

- 추정치에는 산출 로직을 함께 표시한다
- 경쟁사 분석은 STEP 8 토론에서 S5 회의론자의 발언 근거로 활용한다

## STEP 7 — 페르소나 생성 + 확인

### 역할 슬롯 5개

| 슬롯 | 역할 | 핵심 기능 |
|------|------|----------|
| **S1** | 핵심 사용자 | 실제 타겟 대표. 직접적 니즈와 페인포인트 |
| **S2** | 핵심 유사 사용자 | S1과 같은 세그먼트이지만 다른 동기/맥락. S1의 관점을 보완 |
| **S3** | 인접 사용자 | 확장 가능 타겟. "나도 쓸 수 있을까?" 관점 |
| **S4** | 도메인 전문가 | 구조적/산업적 허점 발견. 전문 지식 기반 비평 |
| **S5** | 회의론자 | 악마의 대변인. "왜 굳이?" 질문 담당 |

### 기획 유형별 프리셋

**B2C**: S1 핵심 타겟 | S2 같은 세그먼트 다른 동기 | S3 잠재 사용자 | S4 UX/서비스 전문가 | S5 경쟁 서비스 충성 유저

**B2B**: S1 실무 담당자 | S2 같은 직군 다른 맥락 | S3 구매 결정자/IT 관리자 | S4 업계 애널리스트 | S5 레거시 시스템 만족 유저

**내부 도구**: S1 주요 부서 실무자 | S2 같은 부서 다른 역할 | S3 타 부서/신입 | S4 인프라 담당 | S5 "엑셀이 더 낫다" 파

### 시장 적합성 필터

서비스 특성과 동떨어진 사용자는 자동 제외한다. 예: 온라인 플랫폼이라면 디지털 접근성이 현저히 낮은 60~70대를 페르소나로 설정하지 않는다. 서비스가 실제로 도달할 수 있는 사용자 풀 안에서 다양성을 확보한다.

### 페르소나 깊이 (내부 생성용)

각 페르소나는 3개 레이어로 생성하되, **사용자에게는 요약 테이블만 출력**한다 (Layer 2~3은 토론 시뮬레이션에만 사용):

- **Layer 1 (표면)**: 이름, 나이, 직업, 사용 기기, 일상 루틴
- **Layer 2 (행동)**: 의사결정 스타일, 가격 민감도, 정보 수집 경로, 과거 유사 경험, 신뢰/이탈 트리거
- **Layer 3 (내면)**: 숨은 동기, 내적 갈등, 두려움, 자기 서사

### 화법 차별화

| 페르소나 | 말투 | 논증 방식 |
|---------|------|----------|
| S1 | 일상어, 감정 중심 | "나는 ~할 때 ~해서 불편해요" |
| S2 | 공감 + 비교 | "저도 비슷한데, 저는 오히려 ~" |
| S3 | 호기심 + 조건부 | "~라면 써볼 것 같은데, ~가 걸려요" |
| S4 | 분석적, 데이터 인용 | "업계 표준은 ~인데, 이건 ~" |
| S5 | 도발적, 반문 중심 | "왜?", "그래서?", "근거는?" |

### 갈등 축

페르소나 간 자연스러운 충돌을 위해 최소 3개의 갈등 축을 사전 배치한다:

```
편의성 ◄────────► 안전성/신뢰      (S1 vs S3, S4)
같은 니즈, 다른 해법 ◄────► 확장 가능성  (S1 vs S2 vs S3)
혁신성 ◄────────► 검증된 방식      (S3 vs S5)
```

### 사용자 확인

테이블로 간결하게 표시하고 확인받는다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 7/11 — 페르소나 확인
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 다음 5명으로 토론을 진행합니다.

 | 슬롯 | 이름 (나이) | 한줄 소개 | 핵심 질문 |
 |------|------------|----------|----------|
 | S1 핵심 사용자 | 김지은 (33) 마케터 | "퇴근길이 유일한 내 시간" | "내 하루에 끼어들 수 있나?" |
 | S2 핵심 유사 | 이수진 (31) 기획자 | "점심 고민이지만 이유가 달라" | "같은 문제, 다른 해법은?" |
 | S3 인접 사용자 | 정하늘 (28) 디자이너 | "관심 가면 일단 깔아보는 타입" | "나한테도 해당되나?" |
 | S4 전문가 | 이현수 (41) 기획 10년차 | "숫자로 말해주세요" | "벤치마크 대비 어떤가?" |
 | S5 회의론자 | 최민호 (36) 경쟁앱 유저 | "이미 잘 쓰고 있는 게 있는데" | "왜 굳이 갈아타야 하지?" |

 이대로 진행할까요?
   (a) 진행
   (b) 특정 페르소나 수정 (예: "S3을 20대 대학생으로")
   (c) 페르소나 추가/교체 (예: "투자자 관점 추가")
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 8 — 토론 진행

모든 모드는 **최대 3라운드**로 진행한다.

### 토론 모드

**Standard (기본)**
```
라운드 1: 첫인상 + 쟁점 발굴
라운드 2: 심층 토론 — 핵심 쟁점 2~3개를 깊이 파기
라운드 3: 정리 — 최종 입장 + 합의/미합의 쟁점 명시
```

**Red Team / Blue Team** (`red-blue`)
```
Blue Team (옹호): S1, S2  |  Red Team (공격): S5, S3  |  Judge: S4
라운드 1: Blue 주장 + Red 반박
라운드 2: Blue 방어 + Red 최종 공격
라운드 3: Judge 판정
```

**Pre-mortem** (`premortem`)
```
라운드 1: "이 서비스가 1년 뒤 망했다" — 각자의 사인(死因)
라운드 2: 역추적 + 예방 — "왜 일어났나?" → "뭘 바꾸면 막나?"
라운드 3: 살아남는 시나리오 합의
```

**Six Thinking Hats** (`sixhats`)
```
라운드 1: 흰+빨간 모자 — 사실과 데이터 → 직감과 감정
라운드 2: 검은+노란 모자 — 리스크와 약점 → 기회와 강점
라운드 3: 초록+파란 모자 — 대안과 아이디어 → 종합 판단
```

### 토론 규칙

**충돌 보장**: 매 라운드 최소 2명은 서로 다른 입장을 취한다. 전원 합의 시 반전 카드를 투입한다.

**반전 카드** (토론이 한 방향으로 수렴할 때 투입, 라운드당 최대 1회):
- 경쟁자 등장: "대기업이 동일 기능을 무료로 출시했다면?"
- 규모 변화: "사용자가 100배 늘면 이 구조가 버틸까?"
- 입장 교환: S1과 S5의 입장을 강제 교체
- 제약 추가: "개발 기간이 절반으로 줄었다면?"
- 전제 의심: "이게 진짜 풀어야 할 문제가 맞는가?"

**근거 요구**:
- S1, S2: 개인 경험, 상황, 감정 기반
- S3: 비교 경험, 진입 장벽, 조건부 판단 기반
- S4: 데이터, 업계 사례, 벤치마크 기반
- S5: 반례, 실패 사례, 논리적 허점 기반

**걱정 포인트 반영**: STEP 4의 걱정 포인트는 토론 쟁점으로 반드시 다루되, 뻔한 확인이 아니라 해소 또는 심화까지 도달한다.

### 토론 출력 형식 (토큰 절약 규칙)

- **1인 1발언** — 한 라운드에서 같은 페르소나가 2번 이상 발언하지 않는다
- **발언 2~3문장** — 주장 + 근거 1개. 장황하게 쓰지 않는다
- **동의 발언 생략** — "저도 그렇게 생각해요"는 출력하지 않는다
- **1줄 브릿지** — 라운드 끝에 `▸ 합의: ... | 미합의: ...` 1줄로 전환한다

예시:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 라운드 2: 핵심 쟁점 토론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 쟁점: 온보딩 플로우 — 간소화 vs 충분한 설명

 S1 김지은: "앱 깔고 30초 안에 뭘 할 수 있는지 알고 싶어요.
  튜토리얼 5단계 뜨면 바로 닫아요."

 S2 이수진: "저도 공감하는데, 저는 오히려 점심 추천보다
  같이 먹을 사람 매칭이 더 필요해요."

 S4 이현수: "온보딩 스텝 3개 넘으면 이탈률 급증합니다.
  스텝 수보다 '첫 가치 체험까지의 시간'이 더 중요한 지표예요."

 S5 최민호: "온보딩을 아무리 잘 만들어도 핵심 기능이 직관적이지 않으면
  의미 없어요."

 S3 정하늘: "저는 온보딩 없이 둘러보다가 관심 가면 가입하는 타입이에요."

 [반전 카드: 제약 추가] "온보딩에 1주일밖에 투자할 수 없다면?"
 → S4: "첫 화면 자체가 온보딩이 되게 설계해야 합니다."

 ▸ 합의: 스텝 3개 이하 | 미합의: 가입 전 체험 허용 여부
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 9 — 1:1 심층 인터뷰

토론 완료 후, 사용자에게 1:1 심층 인터뷰 진행 여부를 확인한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 9/11 — 1:1 심층 인터뷰
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 토론이 완료되었습니다.
 특정 페르소나와 1:1 심층 인터뷰를 진행할 수 있습니다.

 토론에서 가장 흥미로웠던 관점:
 • [S? 이름] — [핵심 발언 요약]
 • [S? 이름] — [핵심 발언 요약]

 어떤 페르소나와 인터뷰하시겠어요?
   (a) ~ (e) 각 페르소나
   (f) 건너뛰기 → 보고서로 진행
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

인터뷰 규칙:
- 사용자가 직접 질문하고, 페르소나가 자신의 성격/맥락에 맞게 답변한다
- 토론 때와 동일한 성격, 화법, 내면을 유지한다
- 1:1이므로 "다른 사람 앞에서 말 못 했는데..." 같은 더 솔직한 발언도 자연스럽게 유도한다
- 사용자가 "끝" 또는 "완료"라고 하면 인터뷰를 종료한다

## STEP 10 — 보고서 생성 확인

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 10/11 — 보고서 생성 확인
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 토론과 인터뷰가 완료되었습니다.
 최종 보고서를 생성할까요?

   (a) 보고서 생성 — 전체 분석 보고서 작성
   (b) 요약만 — 핵심 쟁점과 권고사항만 간략히
   (c) 마무리 — 보고서 없이 종료
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

- **(a)** → STEP 11 진행
- **(b)** → 3줄 요약 + 핵심 쟁점 TOP 3 + 즉시 반영 권고사항만 출력
- **(c)** → "토론 내용은 위에 있으니 참고해주세요" 안내 후 종료

## STEP 11 — 보고서 생성

**핵심 7개 섹션**으로 구성한다:

```
┌──────────────────────────────────────────────┐
│  페르소나 토론 보고서                          │
│  [기획 제목] | [토론 모드] | [검증 목적]       │
├──────────────────────────────────────────────┤
│                                              │
│  1. 3줄 요약 + 의사결정 가이드                │
│     토론 결과 3문장 압축                       │
│     + 검증 목적에 대한 직접 답변               │
│     예: "Go, 단 [X] 해결 시"                  │
│                                              │
│  2. 기획 요약 + 시장 분석                     │
│     구조화된 기획 내용                         │
│     + TAM/SAM/SOM + 경쟁 현황 — 표            │
│                                              │
│  3. 토론 하이라이트                            │
│     핵심 쟁점 TOP 3                           │
│     (쟁점별: 대립 요약 + 가장 날카로운 발언 1개)│
│     + 반전 카드 인사이트 (있을 경우)           │
│     + 1:1 심층 인터뷰 핵심 발견 (진행 시)      │
│                                              │
│  4. 걱정 포인트 검증 결과                      │
│     항목별: 결론 + 해소/심화 방안               │
│                                              │
│  5. 기획 평가 + 페르소나 매트릭스              │
│     강점 / 개선점 / 제안 (각 2~3개)           │
│     + 5명 점수표 (관심도/사용의향/지불의향)    │
│                                              │
│  6. 권고사항 (우선순위순)                      │
│     즉시 반영 → 추가 검토 → 장기 과제          │
│     각 항목에 [영향도↑↓ 난이도↑↓] 태그         │
│                                              │
│  7. 예상 Q&A                                 │
│     상사/투자자가 물을 질문 3~5개 + 답변 가이드│
│                                              │
└──────────────────────────────────────────────┘
```

보고서 마지막에 다음 안내를 추가한다:

> 이 결과는 AI 페르소나 기반 시뮬레이션이며, 실제 사용자 테스트를 대체하지 않습니다.

## 후속 액션

보고서 생성 후 사용자에게 다음 액션을 제안한다:

- **특정 페르소나와 추가 인터뷰** — 더 깊이 파고 싶은 관점이 있을 때
- **특정 쟁점 집중 재토론** — 미합의 쟁점을 더 다루고 싶을 때
- **수정된 기획서로 재토론** — v2 기획서와 v1 결과 비교
- **기획서 v2 자동 생성** — 토론 결과를 반영한 개선 기획서
