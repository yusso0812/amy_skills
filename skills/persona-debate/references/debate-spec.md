- 워크플로우 개요

## 워크플로우 개요

- 핵심 경험: "GPT에게 피드백을 요청"하는 느낌보다 "이해관계자 5명이 실제 회의실에서 충돌"하는 느낌을 만든다.
- 핵심 목적: 합의보다 충돌을 통한 발견이다.
- 진행 순서: 11개 STEP을 반드시 순서대로 따른다.
- 사용자 대기: 사용자 응답 또는 확인이 필요한 STEP에서는 멈추고 기다린다.
- 산출물: 토론 로그, 선택형 1:1 인터뷰, 요약 또는 전체 보고서.

## STEP 1. 맥락 질문

다음 형식을 기본으로 사용한다.

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 1/11 — 기획 맥락
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 Q1. 기획 단계가 어디인가요?

   (a) 아이디어 단계 - 방향 자체를 탐색 중
   (b) 구체화 중 - 큰 그림은 있고 디테일 잡는 중
   (c) 거의 확정 - 빠진 거 없는지 최종 점검
   (d) 개발 진행 중 - 피봇/수정 여부 판단

 Q2. 기획 유형은 무엇인가요?

   (a) B2C 서비스/앱
   (b) B2B SaaS/솔루션
   (c) 내부 도구/시스템
   (d) 콘텐츠/미디어
   (e) 하드웨어/IoT
   (f) 기타: ___
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

기획 단계에 따라 기본 토론 모드를 정한다.

| 기획 단계 | 기본 토론 모드 | 보고서 초점 |
|---|---|---|
| 아이디어 단계 | `standard` | 이 방향이 맞는가 |
| 구체화 중 | `standard` | 빠진 게 뭔가 |
| 거의 확정 | `red-blue` | 뚫리는 데가 없는가 |
| 개발 진행 중 | `premortem` | 계속 갈 것인가, 바꿀 것인가 |

기획 유형에 따라 기본 페르소나 프리셋을 정한다.

| 기획 유형 | 프리셋 |
|---|---|
| B2C | `b2c` |
| B2B | `b2b` |
| 내부 도구 | `internal` |
| 기타 | `auto` |

## STEP 2. 기획 내용 입력

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 2/11 — 기획 내용
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 기획 내용을 입력해주세요.

 입력 방법:
   (a) 텍스트로 직접 입력/붙여넣기
   (b) 파일 경로 입력 (md, txt, pdf 지원)

 * 한 줄 아이디어도 괜찮습니다.
   부족한 부분은 뒤에서 보완 질문으로 채웁니다.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

- 한 줄 아이디어부터 상세 문서까지 모두 허용한다.
- 파일 경로가 주어지면 먼저 내용을 읽은 뒤 다음 STEP으로 넘어간다.

## STEP 3. 검증 목적

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 3/11 — 검증 목적
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 이 토론에서 가장 알고 싶은 건 무엇인가요? (복수 선택 가능)

   (a) 만들 가치가 있는가 (시장성/필요성)
   (b) 사용자가 실제로 쓸까 (사용성/UX)
   (c) 돈이 되는 구조인가 (수익성/비즈니스 모델)
   (d) 기술적으로 가능한가 (기술 타당성)
   (e) 뭘 먼저 만들어야 하나 (우선순위)
   (f) 경쟁에서 이길 수 있나 (차별화/포지셔닝)
   (g) 직접 입력: ___
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

선택된 목적은 토론의 초점과 발언 비중에 반영한다.

| 검증 목적 | 강조 질문 | 가중치 힌트 |
|---|---|---|
| 시장성 | 이 문제가 진짜 존재하는가 | S5 강화 |
| 사용성 | 실제 쓰는 장면이 그려지는가 | S1, S2 강화 |
| 수익성 | 돈을 낼 이유가 있는가 | S4, S5 강화 |
| 기술 타당성 | 현실적 제약 안에서 가능한가 | S4 강화 |
| 우선순위 | MVP에 뭘 넣고 뭘 뺄까 | 전원 균등 |
| 차별화 | 왜 기존 것 대신 이걸 쓰나 | S3, S5 강화 |

## STEP 4. 걱정 포인트

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 4/11 — 걱정 포인트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 이미 걱정되는 부분이 있나요? (없으면 건너뛰기)

 예시:
   "수익 모델이 아직 약한 것 같다"
   "타겟이 너무 좁지 않을까"
   "비슷한 서비스가 이미 있는데 차별점이 약하다"

 자유롭게 입력해주세요 (여러 개 가능):
 ___

 (건너뛰려면 Enter)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

처리 규칙:

- 각 걱정 포인트는 토론 쟁점으로 최소 1회 이상 다룬다.
- 단순 확인으로 끝내지 말고, 해소 가능성 또는 더 큰 리스크 여부까지 밀어붙인다.
- 사용자가 건너뛰면 기획 분석을 기반으로 잠재 리스크를 직접 식별한다.

## STEP 5. 기획서 구조화와 보완 질문

기획 내용을 다음 구조로 정규화한다.

```yaml
planning_structure:
  problem:
    target_pain:
    current_alt:
  solution:
    core_feature:
    differentiator:
  target:
    primary_user:
    context:
  business:
    model:
    scale:
  constraints:
    tech:
    resource:
    regulation:
```

운영 규칙:

- 정보가 충분하면 구조화 결과를 보여주고 `"이렇게 이해했습니다. 맞나요?"`로 확인한다.
- 정보가 부족하면 최대 5개까지 보완 질문을 한다.
- STEP 1-4에서 이미 답한 내용은 다시 묻지 않는다.
- 없어도 토론 가능한 항목은 빈칸으로 두고 진행한다.
- 사용자가 `"모르겠다"`고 답하면 그 자체를 토론 쟁점으로 등록한다.

## STEP 6. 시장 규모 분석

사용자 추가 입력 없이 시스템이 자동으로 분석한다.

기본 출력 형식:

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 6/11 — 시장 규모 분석
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 | 구분 | 규모 | 산출 근거 |
 |------|------|----------|
 | TAM | ... | ... |
 | SAM | ... | ... |
 | SOM | ... | ... |

 경쟁: ...
 트렌드: ...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

운영 규칙:

- 추정치는 산출 로직을 함께 적는다.
- 가능하면 최신 자료를 참고하고, 그렇지 못하면 거친 추정임을 밝힌다.
- 경쟁사 분석은 STEP 8에서 S5 회의론자의 근거로 다시 활용한다.

## STEP 7. 페르소나 생성과 확인

항상 5개 슬롯을 사용한다.

| 슬롯 | 역할 | 핵심 기능 |
|---|---|---|
| S1 | 핵심 사용자 | 직접적 니즈와 페인포인트 |
| S2 | 핵심 유사 사용자 | 같은 세그먼트의 다른 동기와 맥락 |
| S3 | 인접 사용자 | 확장 가능 타겟 관점 |
| S4 | 도메인 전문가 | 구조적 허점과 산업 맥락 |
| S5 | 회의론자 | 악마의 대변인 |

프리셋:

- `b2c`: S1 핵심 타겟, S2 같은 세그먼트의 다른 동기, S3 잠재 사용자, S4 UX 또는 서비스 전문가, S5 경쟁 서비스 충성 유저
- `b2b`: S1 실무 담당자, S2 같은 직군의 다른 맥락, S3 구매 결정자 또는 IT 관리자, S4 업계 애널리스트, S5 레거시 시스템 만족 유저
- `internal`: S1 주요 부서 실무자, S2 같은 부서 다른 역할, S3 타 부서 또는 신입, S4 인프라 담당, S5 "엑셀이 더 낫다" 관점
- `auto`: 기획 내용을 보고 가장 현실적인 5명으로 구성

깊이 규칙:

- Layer 1: 이름, 나이, 직업, 사용 기기, 일상 루틴
- Layer 2: 의사결정 스타일, 가격 민감도, 정보 수집 경로, 과거 유사 경험, 신뢰와 이탈 트리거
- Layer 3: 숨은 동기, 내적 갈등, 두려움, 자기 서사
- 사용자에게는 요약 테이블만 보여주고, Layer 2-3은 시뮬레이션 내부에만 유지한다.

화법 규칙:

| 페르소나 | 말투 | 논증 방식 |
|---|---|---|
| S1 | 일상어, 감정 중심 | "나는 ...할 때 ...해서 불편해요" |
| S2 | 공감과 비교 | "저도 비슷한데, 저는 오히려 ..." |
| S3 | 호기심과 조건부 | "...라면 써볼 것 같은데, ...가 걸려요" |
| S4 | 분석적, 데이터 인용 | "업계 표준은 ...인데, 이건 ..." |
| S5 | 도발적, 반문 중심 | "왜?", "그래서?", "근거는?" |

갈등 축을 최소 3개 배치한다.

- 편의성 vs 안전성/신뢰
- 같은 니즈, 다른 해법 vs 확장 가능성
- 혁신성 vs 검증된 방식

시장 적합성 필터:

- 서비스 특성과 동떨어진 사용자는 자동 제외한다.
- 도달 가능한 사용자 풀 안에서만 다양성을 확보한다.

확인용 출력 형식:

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 7/11 — 페르소나 확인
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 다음 5명으로 토론을 진행합니다.

 | 슬롯 | 이름 (나이) | 한줄 소개 | 핵심 질문 |
 |------|------------|----------|----------|
 | S1 핵심 사용자 | ... | ... | ... |
 | S2 핵심 유사 | ... | ... | ... |
 | S3 인접 사용자 | ... | ... | ... |
 | S4 전문가 | ... | ... | ... |
 | S5 회의론자 | ... | ... | ... |

 이대로 진행할까요?
   (a) 진행
   (b) 특정 페르소나 수정
   (c) 페르소나 추가/교체
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 8. 토론 진행

모든 모드는 최대 3라운드로 진행한다.

### 토론 모드

- `standard`
  - 라운드 1: 첫인상과 쟁점 발굴
  - 라운드 2: 핵심 쟁점 2-3개 심층 토론
  - 라운드 3: 최종 입장과 합의/미합의 정리
- `red-blue`
  - Blue Team: S1, S2
  - Red Team: S5, S3
  - Judge: S4
  - 라운드 1: Blue 주장과 Red 반박
  - 라운드 2: Blue 방어와 Red 최종 공격
  - 라운드 3: Judge 판정
- `premortem`
  - 라운드 1: 1년 뒤 실패 가정과 사인
  - 라운드 2: 역추적과 예방책
  - 라운드 3: 살아남는 시나리오 합의
- `sixhats`
  - 라운드 1: 흰모자와 빨간모자
  - 라운드 2: 검은모자와 노란모자
  - 라운드 3: 초록모자와 파란모자

### 토론 규칙

- 충돌 보장: 매 라운드 최소 2명은 다른 입장을 취한다.
- 반전 카드: 토론이 한 방향으로 수렴하면 라운드당 최대 1회 사용한다.

반전 카드 목록:

- 경쟁자 등장: 대기업이 동일 기능을 무료로 출시했다면?
- 규모 변화: 사용자가 100배 늘면 이 구조가 버틸까?
- 입장 교환: S1과 S5의 입장을 강제 교체
- 제약 추가: 개발 기간이 절반으로 줄었다면?
- 전제 의심: 이게 진짜 풀어야 할 문제가 맞는가?

근거 기준:

- S1, S2: 개인 경험, 상황, 감정
- S3: 비교 경험, 진입 장벽, 조건부 판단
- S4: 데이터, 업계 사례, 벤치마크
- S5: 반례, 실패 사례, 논리적 허점

출력 형식 규칙:

- 1인 1발언
- 발언당 2-3문장
- 같은 말 반복이나 단순 동의 발언 생략
- 라운드 끝에 한 줄 브릿지 사용

브릿지 형식:

```text
▸ 합의: ... | 미합의: ...
```

## STEP 9. 1:1 심층 인터뷰

토론 완료 후 다음 형식으로 확인한다.

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 STEP 9/11 — 1:1 심층 인터뷰
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 토론이 완료되었습니다.
 특정 페르소나와 1:1 심층 인터뷰를 진행할 수 있습니다.

 토론에서 가장 흥미로웠던 관점:
 • [S? 이름] — [핵심 발언 요약]
 • [S? 이름] — [핵심 발언 요약]

 어떤 페르소나와 인터뷰하시겠어요?
   (a) ~ (e) 각 페르소나
   (f) 건너뛰기 → 보고서로 진행
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

인터뷰 규칙:

- 사용자가 직접 질문하고 선택된 페르소나만 답한다.
- 토론과 동일한 성격, 화법, 내면을 유지한다.
- 더 솔직한 맥락과 속내가 자연스럽게 나오게 한다.
- 사용자가 `끝` 또는 `완료`라고 하면 인터뷰를 마친다.

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 토론과 인터뷰가 완료되었습니다.
 최종 보고서를 생성할까요?

   (a) 보고서 생성 - 전체 분석 보고서 작성
   (b) 요약만 - 핵심 쟁점과 권고사항만 간략히
   (c) 마무리 - 보고서 없이 종료
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

처리 규칙:

- `(a)`는 STEP 11 전체 보고서로 이어진다.
- `(b)`는 3줄 요약, 핵심 쟁점 TOP 3, 즉시 반영 권고사항만 출력한다.
- `(c)`는 토론 내용을 참고하라는 안내와 함께 종료한다.

## STEP 11. 최종 보고서

전체 보고서는 다음 7개 섹션을 포함한다.

1. 3줄 요약과 의사결정 가이드
2. 기획 요약과 시장 분석
3. 토론 하이라이트
4. 걱정 포인트 검증 결과
5. 기획 평가와 페르소나 매트릭스
6. 권고사항 우선순위
7. 예상 Q&A

세부 구성:

- 1번: 검증 목적에 대한 직접 답과 Go/No-go 또는 조건부 결론
- 2번: 구조화된 기획 요약과 TAM/SAM/SOM, 경쟁 현황
- 3번: 핵심 쟁점 TOP 3, 가장 날카로운 발언, 반전 카드 인사이트, 인터뷰 핵심 발견
- 4번: 걱정 포인트별 결론과 해소 또는 심화 방안
- 5번: 강점, 개선점, 제안 각 2-3개와 5명 점수표
- 6번: 즉시 반영, 추가 검토, 장기 과제로 분류하고 영향도/난이도 태그 표시
- 7번: 상사나 투자자가 물을 질문 3-5개와 답변 가이드

```
