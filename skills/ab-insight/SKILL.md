# AB Insight

> 크몽의 실제 A/B 테스트 기록을 바탕으로, 고민 지점에 대한 근거 있는 방향을 제안한다.

## 입력

사용자 고민: **$ARGUMENTS**

---

## 실행 지침

### Step 1 — A/B 기록 읽기

`/Users/amy/Documents/skills/skills/ab-insight/ab_test_records.md` 파일을 읽는다.
파일이 크므로 전체를 순차 읽기하여 31개 테스트를 모두 파악한다.

### Step 2 — 고민 분류

사용자의 고민을 아래 유형으로 분류한다:

| 유형 | 예시 |
|------|------|
| **UI/UX 결정** | 버튼 위치, 정보 노출 방식, 레이아웃 순서 |
| **기능 추가 여부** | 새 섹션/모듈 추가, 기능 노출 조건 |
| **추천/개인화** | 테마 순서, 콘텐츠 큐레이션 방식 |
| **탐색 구조** | 카테고리 구조, 검색 결과 구성 |
| **전환율 개선** | 문의/구매 퍼널 최적화 |
| **실험 설계** | 지표 선정, 가설 수립, 변수 결정 |

### Step 3 — 관련 테스트 추출

고민과 유사하거나 참고 가능한 테스트를 3개 이내로 추출한다.

선택 기준:
- 동일한 영역(홈/검색/상세/카테고리 등)의 테스트
- 동일한 유형의 의사결정(노출 순서, 버튼, 정보 구조 등)
- 반대 결과(성공/실패)를 모두 포함하여 균형 있는 근거 제공

### Step 4 — 방향성 도출

추출한 테스트 근거를 바탕으로 방향성을 제안한다.

**판단 기준**:
- 과거 테스트에서 같은 방향이 성공했는가?
- 실패한 케이스의 실패 이유가 이번 상황과 다른가?
- 카테고리/플랫폼/유저 세그먼트 조건이 유사한가?

**결론 유형**:

| 결론 | 조건 |
|------|------|
| **방향 명확** | 유사 테스트에서 일관된 결과가 있을 때 |
| **조건부 권장** | 카테고리·플랫폼별로 결과가 달랐을 때 |
| **검증 필요** | 참고할 데이터가 없거나 상충할 때 |
| **주의 필요** | 유사 케이스에서 부정 결과가 있었을 때 |

## 원칙

- 근거 없는 추측은 하지 않는다. 기록된 테스트 외의 일반론은 "기록된 근거 없음"으로 명시한다.
- 성공 케이스만 나열하지 않는다. 실패 케이스와 조건부 결과도 함께 제시한다.
- 수치가 있으면 반드시 포함한다 (예: +8.03%, p < 0.05).
- 카테고리/플랫폼 조건이 다르면 "조건이 다를 수 있음"을 명시한다.
