# AI 퍼소나 유저 테스트

사용자가 제공한 제작물을 분석하고, AI 퍼소나를 생성하여 Think-aloud 유저 테스트를 시뮬레이션한 뒤 UX 리포트를 작성한다.

## 사용법

```
/user-test <URL 또는 Figma 링크 또는 이미지 경로 또는 텍스트 설명>
```

---

## Step 1 — 입력 수집 + 제작물 분석

### 1-1. 입력 수집

사용자 입력 형태를 판별하여 콘텐츠를 수집한다:

| 입력 형태 | 수집 방법 |
|-----------|-----------|
| URL | WebFetch로 HTML 가져옴. 제목, 메타 설명, 네비게이션, 헤딩, CTA, 본문 텍스트 추출 |
| Figma 링크 (`figma.com/...`) | MCP `get_design_context`로 디자인 정보 + `get_screenshot`으로 스크린샷 추출 |
| 이미지/스크린샷 경로 | Read로 이미지를 시각적으로 분석 — UI 구조, 레이아웃, 텍스트, CTA, 네비게이션 파악 |
| 텍스트 설명 | 그대로 분석 대상으로 사용 |

### 1-2. UX 분석

수집한 콘텐츠를 UX 전문가 관점에서 분석하여 다음을 도출한다:

- **도메인**: 서비스 도메인 (예: 이커머스, SaaS, 포트폴리오 등)
- **서비스 요약**: 1~2문장
- **타겟 유저층**: 추정 타겟 사용자군 (구체적으로 — 나이대, 성별 경향, 직업군 등)
- **핵심 기능**: 주요 기능 목록
- **UX 패턴**: 발견된 UX 패턴
- **페이지 구조**: 네비게이션 요소, 주요 섹션, CTA 요소

### 1-3. 사용자 확인

분석 결과를 아래 포맷으로 표시한 뒤 사용자에게 확인을 요청한다:

```
## 제작물 분석 결과

**도메인**: ...
**요약**: ...
**타겟 유저층**: ...
**핵심 기능**: ...
**UX 패턴**: ...
**페이지 구조**:
  - 네비게이션: ...
  - 주요 섹션: ...
  - CTA 요소: ...
```

> "분석 결과를 확인해주세요. 수정하거나 보충할 내용이 있으면 알려주세요."

사용자가 수정 요청 시 반영 후 다시 표시. 승인 시 Step 2로 진행.

---

## Step 2 — 퍼소나 + 태스크 생성

### 2-1. 핵심 퍼소나 방향 확인

전체 퍼소나를 생성하기 전에, 사용자에게 핵심 타겟 유저상을 직접 물어본다.

> "이 서비스의 핵심 사용자는 어떤 사람인가요? 대략적으로라도 좋습니다. (예: 30대 초보 쇼핑몰 사장님, IT 스타트업 PM 등) 정해진 퍼소나가 없는 경우, "알아서 해줘"라고 말씀해 주세요."

사용자 답변을 받은 뒤, 이를 퍼소나 생성의 중심축으로 삼는다. 사용자가 "알아서 해줘" 또는 별다른 의견이 없으면 Step 1의 타겟 유저층 분석 결과를 기반으로 진행한다.

### 2-2. 퍼소나 생성 (4~6명)

확인된 핵심 퍼소나 방향을 기반으로 전체 퍼소나를 생성한다.

**퍼소나 현실성 원칙** (반드시 준수):
- Step 1에서 도출한 **타겟 유저층에 부합하는 퍼소나만** 생성한다
- 예: 20~30대 여성 타겟 쇼핑몰 → 60대 남성 퍼소나 생성 금지
- 예: B2B SaaS → 일반 소비자 퍼소나 생성 금지
- 다양성은 타겟층 **내부**에서 확보한다

**다양성 요구사항** (타겟층 범위 내에서):
- 나이대: 타겟층 내에서 다양하게 분포
- 기술 숙련도: beginner / intermediate / advanced 골고루
- 사용 맥락: 목적 지향 사용자 + 탐색 지향 사용자 모두 포함
- 비판적 관점: 파워 유저(효율 중시) 또는 회의적 사용자(신뢰 부족) 최소 1명 포함

**최소 구성**:
- 핵심 타겟 유저 2명 이상
- 경계 유저 (타겟 범위의 가장자리) 1명
- 파워 유저 / 효율 중시형 1명 (다수 플랫폼 경험자, 비교 기준이 높고 비효율에 민감)
- 회의적 / 신뢰 부족형 1명 (온라인 서비스에 불신, 신뢰 근거와 투명성을 강하게 요구)

각 퍼소나에 포함할 정보:
- 이름, 나이, 직업
- 기술 숙련도 (beginner / intermediate / advanced)
- 사용 맥락 (이 서비스를 사용하는 상황)
- 성격 특성
- 목표 (이 서비스에서 달성하려는 것)
- 예상 불편사항
- 접근성 요구사항 (해당 시)
- 기기 (desktop / mobile / tablet)
- 대표 발언 (이 퍼소나를 요약하는 한마디)

### 2-2. 퍼소나 사용자 확인

퍼소나만 먼저 표시한 뒤 확인을 요청한다:

```
## 퍼소나

### 1. [이름] ([나이]세, [직업])
- 기술 숙련도: ...
- 사용 맥락: ...
- 목표: ...
- 불편사항: ...
- 대표 발언: "..."

### 2. ...
```

> "퍼소나를 확인해주세요. 수정할 내용이 있으면 알려주세요."

사용자가 수정 요청 시 반영 후 다시 표시. 승인 시 2-3으로 진행.

### 2-3. 테스트 태스크 생성 (3개)

확정된 퍼소나를 기반으로 다음 3가지 태스크를 자동 생성한다:
1. **메인 페이지 탐색**: 주요 콘텐츠/기능을 찾아보는 태스크
2. **CTA 상호작용**: 핵심 행동 유도 요소를 사용하는 태스크
3. **네비게이션 활용**: 원하는 정보를 찾아 이동하는 태스크

각 태스크에는 **설명**과 **성공 기준**을 포함한다.

### 2-4. 태스크 사용자 확인

태스크를 표시한 뒤 확인을 요청한다:

```
## 테스트 태스크

1. **[태스크명]**: [설명] (성공 기준: ...)
2. ...
3. ...
```

> "테스트 태스크를 확인해주세요. 수정할 내용이 있으면 알려주세요."

사용자가 수정 요청 시 반영 후 다시 표시. 승인 시 Step 3으로 진행.

---

## Step 3 — 테스트 시뮬레이션 + 리포트

### 3-1. Think-aloud 시뮬레이션

확정된 퍼소나 전원에 대해 **일괄로** 테스트 시뮬레이션을 수행하되, **요약 테이블 형태**로 출력한다.

#### 출력 포맷

**퍼소나별 요약 테이블** (퍼소나당 1개 테이블):

```
### [이름] ([나이]세, [직업]) — [유형 태그]

| 태스크 | 결과 | 난이도 | 핵심 행동 요약 | 주요 감정 흐름 | 발견 이슈 |
|--------|------|--------|----------------|----------------|-----------|
| 태스크1 | ✅/❌ | 2/5 | 행동1 → 행동2 → 행동3 | 😊→😐→😕 | [major] 이슈 설명 |
| 태스크2 | ... | ... | ... | ... | ... |
| 태스크3 | ... | ... | ... | ... | ... |

> **전체 소감**: "퍼소나 관점의 한줄 소감"
```

각 셀의 작성 기준:
- **핵심 행동 요약**: 3~5단계를 화살표(→)로 연결한 한 줄 요약
- **주요 감정 흐름**: 😊(positive) 😐(neutral) 😕(confused) 😤(frustrated) 이모지로 흐름 표현
- **발견 이슈**: [심각도] 이슈 한 줄 설명. 복수 이슈 시 줄바꿈으로 구분

시뮬레이션 내부에서는 퍼소나별·태스크별로 다음을 산출하되, 위 테이블에 요약하여 출력한다:
- 행동 단계, 태스크 성공 여부, 난이도(1~5), 예상 소요 시간
- 발견된 UX 이슈: 심각도(critical/major/minor), 카테고리(usability/accessibility/clarity/efficiency/visual), 설명, 위치, 개선 제안

**시뮬레이션 핵심 원칙**:
- 퍼소나의 기술 숙련도, 나이, 성격을 행동 패턴에 반영해야 한다
- 초보자는 더 많은 혼란 단계를, 파워 유저는 효율성 문제를 더 많이 발견한다
- 파워 유저는 효율성·정보 밀도·비교 기능 부재 이슈를 집중 발견한다
- 회의적 사용자는 신뢰 요소 부족(후기 수, 평점, 환불 정책, 자격 검증 등) 이슈를 집중 발견한다

### 3-2. 이슈 클러스터링

모든 퍼소나의 이슈를 수집한 뒤 클러스터링한다:

- **동일 카테고리 + 동일 위치**의 이슈를 그룹화
- 3명 이상 보고한 이슈 → severity를 **critical**로 격상
- 2명 이상 보고한 minor 이슈 → **major**로 격상
- 중복 이슈는 "(N명의 퍼소나가 동일 문제를 보고함)" 표기

### 3-3. 리포트 작성

시뮬레이션 결과를 아래 포맷으로 종합한다:

```
## UX 테스트 리포트

### Executive Summary
(2~3문단: 주요 발견사항, 핵심 인사이트, 전반적 평가)

### UX 점수
| 항목 | 점수 |
|------|------|
| 종합 | /100 |
| 사용성 | /100 |
| 접근성 | /100 |
| 직관성 | /100 |
| 효율성 | /100 |

### 발견사항

#### Critical
- [이슈 설명] — 위치: ... | 카테고리: ... | 개선 제안: ...

#### Major
- ...

#### Minor
- ...

### 퍼소나별 결과 요약
| 퍼소나 | 태스크1 | 태스크2 | 태스크3 | 전체 소감 |
|--------|---------|---------|---------|-----------|
| [이름] | 성공/실패 (난이도) | ... | ... | 한줄 요약 |

### 개선 권고안

#### 높은 우선순위
- **[제안 제목]**: [상세 설명] (예상 효과: ...)

#### 중간 우선순위
- ...

#### 낮은 우선순위
- ...
```

**권고안 우선순위 기준**:
- **high**: 태스크 실패를 유발하거나 3명 이상 보고한 문제
- **medium**: 사용성 점수를 크게 낮추거나 2명 이상 보고한 문제
- **low**: 일부 퍼소나에게 불편하지만 기능에 큰 영향 없는 문제
